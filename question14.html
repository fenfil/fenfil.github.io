Вопрос 14. Как исправить описание любого коммита в истории

<р> Описанные ниже действия возможны, если коммиты еще не отправлены на удаленный репозитарий, или если программист является единственным пользователем удаленного репозитария. Описанные ниже действия являются переписыванием истории. Начиная с git 1.6.6 можно использовать механизм GIT, называющийся reword. Он используется в рамках команды rebase. Для начала находим ID коммита с неправильным комментарием. На выходе получаем список коммитов с хеш-идентификаторами. git откроет редактор по умолчанию. Это сценарий действий, которые будут выполнены для изменения истории. GIT предложит сохранить текстовый файл, что и нужно сделать. Фактически, в интерактивном режиме, это список команд для него. Он снова откроет текстовый редактор и предложит изменить комментарий для нужного коммита. После сохранения файлика git повторит весь комит, поменяет его ID и короткий хеш. Все. Кстати, помимо reword есть еще и команда edit. После сохранения файла сценария, сценарий будет выполняться до команды edit, после чего GIT завершит работу и вернется в консоль В этот момент рабочая директория будет находиться в состоянии коммита c4d25d6. Можно изменить это состояние. Откроется редактор, в котором можно будет изменить описание коммита. После закрытия редактора, отредактированный коммит будет изменен в истории. Эта команда будет дальше искать коммиты, которые нужно изменить, и будет снова останавливаться чтобы позволить внести изменения, до самого последнем коммите. После нее история на origin-сервере будет перезаписана. И если с репозитарием работают другие программисты, они должны будут принять всю измененную историю. Как это сделать - надо искать отдельно. Радикальный способ - это сделать git clone в отдельной директории и продолжить работу в ней.
